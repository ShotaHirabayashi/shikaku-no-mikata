## 問1

ソフトウェア開発において、要件定義→設計→実装→テスト→運用の各工程を順番に進め、原則として前の工程に戻らない開発モデルはどれか。

1. スパイラルモデル
2. プロトタイピングモデル
3. ウォータフォールモデル
4. アジャイル開発モデル


<QuizAnswer>

**答え: 3**

ウォータフォールモデルは、上流工程から下流工程へ滝の水が流れ落ちるように順番に進める開発モデルです。各工程の成果物を確認してから次の工程に進むため、大規模システムの開発に適しています。原則として前工程への後戻りをしない点が特徴です。

</QuizAnswer>

## 問2

アジャイル開発手法の一つであるスクラムにおいて、1〜4週間程度の短い期間で開発サイクルを繰り返す単位を何と呼ぶか。

1. マイルストーン
2. イテレーション（スプリント）
3. フェーズ
4. ベースライン


<QuizAnswer>

**答え: 2**

スクラムでは、スプリント（イテレーション）と呼ばれる1〜4週間程度の短い反復期間を設定し、その中で計画・設計・実装・テストを繰り返します。各スプリントの終了時に動作するソフトウェアを提供することで、変化する要求に柔軟に対応できます。

</QuizAnswer>

## 問3

プロジェクトマネジメントにおいて、作業を階層的に分解し、プロジェクト全体の作業範囲を明確にするための手法はどれか。

1. ガントチャート
2. PERT図
3. WBS（Work Breakdown Structure）
4. アローダイアグラム


<QuizAnswer>

**答え: 3**

WBS（Work Breakdown Structure：作業分解構成図）は、プロジェクトの成果物や作業を階層的に細分化し、管理可能な単位に分解する手法です。WBSを作成することで、プロジェクトのスコープ（範囲）を明確にし、必要な作業の漏れを防ぐことができます。ガントチャートやPERT図はスケジュール管理に使われる手法です。

</QuizAnswer>

## 問4

ITサービスマネジメントにおけるITILの考え方で、利用者からの問合せや障害報告を一元的に受け付ける窓口を何と呼ぶか。

1. コールセンタ
2. ヘルプデスク
3. サービスデスク
4. カスタマサポート


<QuizAnswer>

**答え: 3**

サービスデスクは、ITILにおいて利用者と IT サービス提供者の間の単一の窓口（SPOC: Single Point of Contact）として機能する組織です。障害報告、問合せ、サービス要求などを一元的に受け付け、適切な対応部門への振り分けや対応状況の管理を行います。ヘルプデスクと混同されやすいですが、ITILではサービスデスクという用語が正式に使われます。

</QuizAnswer>

## 問5

SLA（Service Level Agreement）の説明として、最も適切なものはどれか。

1. システム開発の契約時に取り交わす秘密保持契約
2. ITサービスの提供者と利用者の間で合意するサービス品質の水準に関する取り決め
3. ソフトウェアの使用許諾に関するライセンス契約
4. プロジェクトの成果物を定義するスコープ記述書


<QuizAnswer>

**答え: 2**

SLA（Service Level Agreement：サービスレベル合意書）は、ITサービスの提供者と利用者の間で、提供するサービスの内容や品質水準（可用性、応答時間、障害復旧時間など）を具体的な数値目標として合意した文書です。SLAを定めることで、サービス品質の維持・改善を客観的に管理できます。

</QuizAnswer>

## 問6

システム監査において、監査対象から独立した立場で、情報システムの信頼性・安全性・効率性などを検証・評価する人を何と呼ぶか。

1. プロジェクトマネージャ
2. システムアナリスト
3. システム監査人
4. 情報セキュリティ管理者


<QuizAnswer>

**答え: 3**

システム監査人は、監査対象の組織や部門から独立した客観的な立場で、情報システムの信頼性・安全性・効率性を総合的に検証・評価する専門家です。この独立性の確保はシステム監査の最も重要な原則の一つであり、監査結果の客観性と信頼性を保証するために必要不可欠です。

</QuizAnswer>

## 問7

プロジェクトの各作業の依存関係と所要日数を矢印で表現し、プロジェクト全体の最短完了日数や最も余裕のない経路（クリティカルパス）を求める手法はどれか。

1. WBS
2. パレート図
3. アローダイアグラム
4. 管理図


<QuizAnswer>

**答え: 3**

アローダイアグラムは、作業（アクティビティ）を矢印で、結合点をノード（○）で表し、作業間の順序関係と所要時間を図示する手法です。この図からクリティカルパス（最長経路＝余裕が全くない作業の連鎖）を求めることで、プロジェクト全体の最短完了日数を把握できます。PERT（Program Evaluation and Review Technique）とも関連が深い手法です。

</QuizAnswer>

## 問8

ITサービスマネジメントにおいて、サービスの中断（インシデント）が発生した際に、根本原因の究明よりも、まずサービスの迅速な復旧を優先して対応するプロセスはどれか。

1. 問題管理
2. 変更管理
3. インシデント管理
4. リリース管理


<QuizAnswer>

**答え: 3**

インシデント管理は、ITサービスの中断や品質低下が発生した際に、可能な限り迅速にサービスを復旧させることを目的としたプロセスです。根本原因の特定と恒久的な対策は「問題管理」プロセスで行います。インシデント管理では、暫定的な回避策であってもまずサービスを正常な状態に戻すことを最優先とします。

</QuizAnswer>

## 問9

ソフトウェアのテストにおいて、モジュール（部品）単位のテストが完了した後、モジュール同士を組み合わせて、モジュール間のインタフェースが正しく機能するかを確認するテストはどれか。

1. 単体テスト
2. 結合テスト
3. システムテスト
4. 運用テスト


<QuizAnswer>

**答え: 2**

結合テスト（統合テスト）は、単体テストで個々の動作が確認されたモジュールを組み合わせ、モジュール間のデータの受け渡しやインタフェースが正しく機能するかを検証するテストです。単体テストの次の段階で行われ、この後にシステム全体の要件を確認するシステムテスト、実際の運用環境で確認する運用テストへと進みます。

</QuizAnswer>

## 問10

プロジェクトマネジメントにおいて、プロジェクトの作業範囲（何を含み、何を含まないか）を明確に定義・管理する活動を何と呼ぶか。

1. コスト管理
2. 品質管理
3. スコープ管理
4. リスク管理


<QuizAnswer>

**答え: 3**

スコープ管理は、プロジェクトで実施すべき作業の範囲（スコープ）を明確にし、過不足なく管理する活動です。スコープが曖昧なままプロジェクトを進めると、作業範囲が際限なく拡大する「スコープクリープ」が発生し、コスト超過や納期遅延の原因になります。WBSの作成はスコープ管理の重要な手段の一つです。

</QuizAnswer>
